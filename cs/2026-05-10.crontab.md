# crontab 定时任务

## 什么是 crontab

crontab 是 Unix/Linux 系统下用于设置**周期性定时任务**的工具。`cron` 是一个后台守护进程（daemon），负责在指定的时间点自动执行命令；`crontab`（cron table）则是用于管理这份"定时任务清单"的命令。

常见使用场景：

- 定时备份数据库
- 定时清理日志文件
- 定时同步数据或抓取接口
- 定时执行脚本（如发送邮件、统计报表）

---

## crontab 命令

```bash
crontab -e   # 编辑当前用户的定时任务
crontab -l   # 列出当前用户的所有定时任务
crontab -r   # 删除当前用户的所有定时任务
crontab -u <user> -l  # 查看指定用户的定时任务（需 root 权限）
```

---

## 时间表达式语法

每一行 crontab 记录由**6个字段**组成（五个时间字段 + 一个命令字段）：

```
┌─────────── 分钟 (0–59)
│  ┌────────── 小时 (0–23)
│  │  ┌───────── 日期 (1–31)
│  │  │  ┌──────── 月份 (1–12)
│  │  │  │  ┌─────── 星期 (0–7，0 和 7 均代表周日)
│  │  │  │  │
*  *  *  *  *  command
```

### 特殊字符

| 字符 | 含义       | 示例                                        |
| ---- | ---------- | ------------------------------------------- |
| `*`  | 任意值     | `* * * * *` 每分钟执行                      |
| `,`  | 枚举多个值 | `1,15,30 * * * *` 每小时的第 1、15、30 分钟 |
| `-`  | 范围       | `0 9-18 * * *` 9点到18点整点执行            |
| `/`  | 步长       | `*/5 * * * *` 每 5 分钟执行一次             |

## 常用示例

```bash
# 每分钟执行一次
* * * * * /path/to/script.sh

# 每天凌晨 2:30 执行备份
30 2 * * * /usr/local/bin/backup.sh

# 每周一早上 8:00 发送周报
0 8 * * 1 /usr/local/bin/weekly_report.sh

# 每月 1 日零点清理日志
0 0 1 * * /usr/local/bin/clean_logs.sh

# 每 5 分钟检查一次服务状态
*/5 * * * * /usr/local/bin/check_service.sh

# 工作日（周一到周五）每天 9:00–18:00 每小时执行
0 9-18 * * 1-5 /usr/local/bin/job.sh

# 每天 0 点和 12 点各执行一次
0 0,12 * * * /usr/local/bin/job.sh
```

## 环境变量问题

cron 运行时使用的是**最小化的 shell 环境**，与交互式登录 shell 不同，`PATH`、`NODE_ENV` 等环境变量均不会自动加载。这是 crontab 新手最常见的坑。

**解决方法 1：在 crontab 文件顶部声明环境变量**

```bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
HOME=/root

0 2 * * * /usr/local/bin/backup.sh
```

**解决方法 2：在脚本内 source 相应的配置文件**

```bash
#!/bin/bash
source /etc/profile
source ~/.bashrc
# ... 正式逻辑
```

**解决方法 3：用绝对路径调用命令**

```bash
# 不要写
0 2 * * * node /path/to/app.js

# 改为
0 2 * * * /usr/local/bin/node /path/to/app.js
```

## 输出与日志

cron 默认将命令的 stdout/stderr 通过邮件发给执行用户。如果不需要邮件通知，通常会将输出重定向到日志文件：

```bash
# 只记录 stdout
*/5 * * * * /usr/local/bin/job.sh >> /var/log/job.log 2>&1

# 完全丢弃输出
*/5 * * * * /usr/local/bin/job.sh > /dev/null 2>&1
```

> `2>&1` 表示将 stderr 合并到 stdout，确保错误信息也写入日志。

## macOS 上的注意事项

macOS 同样支持 crontab，但苹果官方推荐使用 **launchd**（通过 `.plist` 文件管理定时任务）替代 cron。不过 crontab 在 macOS 上依然可用，安装 Homebrew 后也可以通过它管理的 cron 守护进程。

macOS 上还需注意**全磁盘访问权限**：如果脚本需要读写受保护的目录（如 Desktop、Documents），需要在「系统设置 → 隐私与安全性 → 完全磁盘访问权限」中为 `/usr/sbin/cron` 开放权限，否则脚本会静默失败。

### 查看当前所有定时任务

macOS 上的定时任务来源有三处，需要分别查看：

**1. crontab（当前用户）**

```bash
crontab -l
```

**2. launchd — plist 文件**

launchd 是 macOS 的任务调度核心，plist 存放在以下目录：

| 目录                             | 说明                                   |
| -------------------------------- | -------------------------------------- |
| `~/Library/LaunchAgents/`        | 当前用户登录后运行的 Agent             |
| `/Library/LaunchAgents/`         | 所有用户登录后运行的 Agent             |
| `/Library/LaunchDaemons/`        | 系统级守护进程（开机即运行，无需登录） |
| `/System/Library/LaunchAgents/`  | macOS 系统自带 Agent（勿改动）         |
| `/System/Library/LaunchDaemons/` | macOS 系统自带守护进程（勿改动）       |

列出当前用户的 LaunchAgents：

```bash
ls ~/Library/LaunchAgents/
ls /Library/LaunchAgents/
ls /Library/LaunchDaemons/
```

查看某个 plist 的内容：

```bash
cat ~/Library/LaunchAgents/com.example.myjob.plist
```

**3. launchctl 命令——查看已加载的任务**

```bash
# 列出当前用户已加载的所有服务（含系统服务）
launchctl list

# 过滤只看自定义的（排除 com.apple 开头的系统项）
launchctl list | grep -v com.apple
```

输出三列含义：`PID`（运行中为进程号，未运行为 `-`）、`最近退出码`、`Label`（任务名）。

### plist 文件结构示例

下面是一个每天 2:30 执行备份脚本的 plist：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.example.backup</string>

    <key>ProgramArguments</key>
    <array>
        <string>/usr/local/bin/backup.sh</string>
    </array>

    <!-- 每天 02:30 执行 -->
    <key>StartCalendarInterval</key>
    <dict>
        <key>Hour</key>
        <integer>2</integer>
        <key>Minute</key>
        <integer>30</integer>
    </dict>

    <key>StandardOutPath</key>
    <string>/tmp/backup.log</string>
    <key>StandardErrorPath</key>
    <string>/tmp/backup.err</string>
</dict>
</plist>
```

保存到 `~/Library/LaunchAgents/com.example.backup.plist` 后，用 launchctl 加载：

```bash
# 加载（立即生效）
launchctl load ~/Library/LaunchAgents/com.example.backup.plist

# 卸载
launchctl unload ~/Library/LaunchAgents/com.example.backup.plist

# 手动触发一次（调试用）
launchctl start com.example.backup
```

### crontab vs launchd 对比

| 维度     | crontab                    | launchd (plist)                   |
| -------- | -------------------------- | --------------------------------- |
| 配置方式 | 文本表达式                 | XML 文件                          |
| 最小粒度 | 1 分钟                     | 1 秒                              |
| 开机自启 | 需登录后 cron 守护进程启动 | LaunchDaemons 开机即启动          |
| 日志支持 | 需手动重定向               | plist 直接配置 stdout/stderr 路径 |
| 官方推荐 | 否（macOS）                | 是                                |
| 适合场景 | 快速配置、跨平台脚本       | macOS 原生、需要精细控制          |

## 小结

| 场景         | 推荐写法         |
| ------------ | ---------------- |
| 每 N 分钟    | `*/N * * * *`    |
| 每天固定时间 | `分 时 * * *`    |
| 每周固定     | `分 时 * * 星期` |
| 每月固定     | `分 时 日 * *`   |

crontab 简单易用，适合大多数定期轻量任务。对于依赖项复杂、需要监控和重试机制的任务，可以考虑引入 [node-cron](https://github.com/node-cron/node-cron)、Celery、或云平台的定时任务服务（如 AWS EventBridge、阿里云定时触发器）。
