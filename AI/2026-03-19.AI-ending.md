## 传统的软件售卖逻辑变化

- 中国地区：软件免费，部分高级功能付费，或者用平台广告和生态经济来赚钱。比如微信免费用，但是微信支付手续费千3，千6。公众号免费看，但是服务号每年交个服务费
- 海外：SaaS(software as a service)，各种软件按月付费，或者按年付费，或者按功能付费

但是AI时代，熟悉软件不再是单纯的工具，而是一个智能体，能够自主完成任务，甚至创造新的功能。这就引入了一个新的概念：** GaaS（agent as a service）**

你可以把AI Agent看作是一个智能的服务提供者，它不仅提供功能，还能根据用户的需求进行定制和优化。用户不再需要关心底层的模型和技术细节，只需要关注Agent能否满足他们的需求

那传统软件还有存在的必要吗？短期可能继续存在，因为用户习惯不会一下子适应Agent。用户可能习惯了MicroSoft Excel公式怎么写，Word软件的加粗按钮在哪，这种习惯的转移需要时间。

但是软件的概念将逐渐被Agent所取代。用户不再购买传统软件，而是订阅Agent服务，根据自己的需求选择不同的Agent来完成任务

**Agent as a service = Agent Interface + Agent Engine + Agent Ecosystem**

目前市场上Agent Interface + Agent Engine 都很多，但是 Agent Ecosystem 还在赛马。并没有一个统一的 Agent Ecosystem 可以干完所有的事

传统的软件是否提供MCP和Skill也还在选择中，担心使用软件的用户不适应，或者担心开放了能力被Agent取代

关于传统软件与Agent的关系，可以看看MicroSoft的选择。它推出了 [Copilot Cowork](https://www.microsoft.com/en-us/microsoft-365/blog/2026/03/09/copilot-cowork-a-new-way-of-getting-work-done/)，在Office软件中集成了Copilot功能，让用户可以通过自然语言与软件交互，完成各种任务。这种方式既保留了传统软件的界面和功能，又引入了Agent的智能能力，提供了更好的用户体验


## Token 货币化

当人类习惯了Agent的便利性和智能化，由模型+数据中心+电力生产的Token就逐渐成为了新的货币形式。用户通过购买token来使用Agent的服务，根据不同的需求和使用频率选择不同的token套餐。

因为越多的token消耗和资源投入，也就是上下文越多，推理能力越强，Agent就会变得越智能

```bash
花钱可以变得更强 = 花token = 拥有能力
```

比如我是一个做旅游行业的运营同学，我需要一个报表Agent帮我分析手头上收集的大量行业数据，生成报表。我可以购买一个专业旅游行业Agent的套餐，根据我的需求选择不同的功能和服务。这个专业旅游行业Agent可以根据我的数据分析需求，自动生成各种类型的报表，甚至基于全网数据提供不同国家地区，不同社交平台抓取最热门的旅游话题，给出数据分析结论和后续建议

## Token生产商的困局
Token依赖GPU等硬件+电力，Navida 作为GPU等硬件供应商，Vera Rubin架构的提出者。股价近期却一直在 **$180** 附近徘徊，华尔街担心数据中心建设硬件支出不能收回利润，且英伟达硬件换代过快客户持续支出

关键看各大云厂商资本支出增速和英伟达推理收入占比，只要Agent能卖出去，Token就有市场，GPU等硬件的需求也会持续增长

## 人与AI的协同

### 输入输出速率差异
人脑能处理的数据，跟后面AI Agent能处理的数据，会越拉越长。如果假设人类速率恒定，那么AI可以通过更大的Context，更多的参数数量，更多的硬件堆叠来提升输入输出性能

** AI时代，人类使用Agent输出很快，但是人类输入很慢。瓶颈在人不在机器，机器和流程是可优化的 **

### 并行多Agent协同
现在已经是一个人操控多个AI并行的环境了，比如你开了ChatGPT，Gemini，Claude等多个Agent在同时处理不同的任务。未来可能会有更多的Agent出现，用户需要同时管理和协调这些Agent来完成各种任务。

但是 **并行碎片化任务会极大分割个人的精力和体力**, 正如AI在不同上下文切换的效率很差，人类也是如此。不同的工作上下文切换需要适应和锻炼

或者换句话说：**人类的身体和大脑跟不上AI Agent对人类的要求**


### 工具迭代

对于某类任务，不同的模型&工具短时间有一些优势。但是2026目前处于AI应用大规模爆发阶段，所以工具迭代会非常快，可能几个月就有一个新的工具出现，或者现有工具的能力大幅提升。用户需要不断适应和学习新的工具来满足自己的需求

## 工程角度讨论

### AI生成代码用户生产环境是否长期可行？

- 很多AI生成的代码如果不加管控。长期系统腐化严重，出了bug难排查
- **如果进行一定的管控来长期运行，需要前期对AI生成的代码进行架构设计，完善测试等流程。应该可以一定程度延缓代码腐化的速度, 这也是harness engineer的工作内容之一**
- 核心模块必须多重review，不能完全依赖AI生成的代码
- 如果是月抛项目, 可能不需要太担心代码腐化问题，毕竟项目周期短，后续维护也不多
